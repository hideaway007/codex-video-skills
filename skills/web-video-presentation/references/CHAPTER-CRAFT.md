# 章节开发指引（每章开发必读）

---

## 这是视频，不是 PPT

正在做的是**视频网页** —— 讲者点击 + 口播 + 录屏发出去给观众看。
判断每一步做对没有，标准非常朴素：

- **不像 PPT** —— 观众感觉是在看视频，不是在看翻页幻灯（页面中不得包含页眉页脚，突出主视觉元素）
- **看起来舒服** —— 配色、字体、节奏都让人放松，不得出现大量的纯文字、不得出现字体太小的文字
- **有视觉冲击** —— 画面在演事情，不只是文字堆砌，不得一次性全部罗列所有元素，关键元素随进度逐步推进展现

---

## 必须用 CSS / SVG / Canvas / JS 大胆绘制视觉演示

> **这是底线。**
>
> 每一章都至少要有 1~2 处"动起来的图 / 演示元素"。
> **整章只有纯文字 = 验收不过 = 回去重做。**

视频感最强的来源 —— 用户**看见**了被讲解的东西在屏幕上演给他看：

- 数字在递增 / 横条在生长 / 排名在交换
- 流程节点依次点亮 / 连线自绘
- 对比被一刀切开 / 聚光灯扫过 / 形状在变形
- 粒子聚拢成形 / 噪声背景流动 / 字符雨下落
- 模拟终端交互
- 模拟 AI 对话窗口
- 模拟文件目录树

**怎么组合发挥都行 —— 但每章必须用，不允许整章纯文字。**

进一步要求：每章至少要有一个“富 HTML 主视觉”承担解释任务。优先用本章内
React 小组件 + SVG / CSS / Canvas 做出来，而不是把文字放进卡片。

可选原语包括：

- SVG 图表：bar、stacked bar、donut、gauge、sparkline、radar、path flow
- 信息组件：metric card、status dot、verdict tag、progress ledger
- 模拟界面：browser mock、terminal mock、file tree、chat bubbles、dashboard
- 结构演示：timeline、flowchart、before-after split、node map、document wall

具体选型参考 [`RICH-HTML-TOOLKIT.md`](RICH-HTML-TOOLKIT.md)。不要把它当模板
照抄；先判断本章内容关系，再选 1 个主视觉 + 1 个辅助信息组件。

---

## 默认优先：SVG 分镜小剧场（CSS 档）

遇到概念旅程、工具链、输入输出、流程、演化、对比、观点推进时，默认先考虑
做成 **SVG 分镜小剧场**：画面像一个可录屏的微型解释场景，而不是一张静态
插画或一组卡片。

- **角色必须参与解释**：优先加 1 个简单小人 / 讲解者 / 操作者，用指向、
  行走、观察、递交文档、看浏览器等动作帮助观众理解关系。小人不需要复杂
  骨骼，CSS transform / path reveal / opacity 就够。
- **镜头不是只等元素出现**：页面要有轻微 pan / zoom / slide camera。
  例如从“想法”推到 Markdown，再滑到 HTML，最后收束到 Video + Voice。
  镜头变化由外层 `.scene-camera` 的 CSS transform 跟 step 同步控制。
- **SVG 自绘要明显**：路径线、窗口边框、波形、图表线条优先用
  `stroke-dasharray` + `stroke-dashoffset` 一段段画出来；不要只做 fade in。
- **节点状态要有三态**：当前节点为 `active`（放大、加粗、亮起），前序节点为
  `done`（保留但降低对比，作为上下文），后续节点为 `muted`（半透明等待）。
- **局部动画跟 step 同步**：Markdown 文档行数增长、HTML 窗口小图表生长、
  Video 节点声波出现、底部 bandwidth / progress 条推进，都应该跟当前 step
  对齐，而不是全部循环播放。
- **多层 SVG 组织画面**：背景层放淡纸面网格、噪声纹理或手绘云线；中景层放
  流程路径、节点和连接关系；前景层放人物、文档、浏览器、视频节点和字幕焦点。
  层次越清楚，越像网页视频，而不是单张配图。
- **默认不上 JS / IK**：CSS keyframes、transition、SVG stroke animation 和
  step class 能完成时保持轻量。只有用户明确要自行车踩踏、机械联动、骨骼反解
  这类连续运动真实性，才考虑 JS / IK。

## 逐步揭示，禁止一次全展示

整页内容由**全局 `step` 计数器**驱动 —— 点击空白处或按 → 键推进
一步。设计每一步时心里要默念：**这一步演什么，下一步演什么**。

**最重要的一条**：

> 当口播在说"第一是 X、第二是 Y、第三是 Z"这种**清单 / 列表**时，
> **严禁**一个 step 把 X / Y / Z 全部 stagger 上来。

正确做法：

- 一项 = 一个 step
- X 只在它自己的 step 里独自亮起
- 讲到 Y 时，X 灰化保留作上下文 + Y 亮起
- 讲到 Z 时，X / Y 都灰化 + Z 亮起

**判断标准**：讲者会一个一个念出来吗？会 → 必须逐个揭示。

---

## 内容取舍：抓重点，不要原文搬运

视频是**音 + 画**：

- **口播**负责把信息线性讲清楚
- **画面**负责把节拍重点放大、节奏感拉出来

每个 step 屏幕上只挂这个节拍**最值得放大的 1~3 个东西** —— 一个
hero 标语 / 一个数字 / 一组对比 + 必要的视觉演示。

不要试图把原文每个字都搬上去。那是论文阅读，不是视频。

---

## 双源：节奏跟口播稿，细节回原文章

> **节奏 / 顺序 / 节拍切分** 跟 **`script.md` 口播稿** —— **关键顺序不能乱**。
> **画面细节 / 数据 / 引用 / 案例** 回 **`article.md` 原文章**抽。

`outline.md` 已经在每章首段抽了「信息池」做参考。但**实现章节时
也必须回去翻 `article.md` 本章对应段落** —— 那里有比口播稿多得多的
细节（具体数字、引用原话、案例维度、出处时间）。把这些挂到画面上，
让**画面信息密度 > 口播信息密度**。

> **如果你只用了口播稿的内容做章节** —— 屏幕等于把口播打字打了一遍
> —— 那就是 PPT，不是视频。
>
> 章节实现一定要回原始文章抽细节，不要嫌麻烦。

## 字体 / 配色 / 动画 / 留白 —— 视频演示基本审美

视频观众离屏幕远、注意力浮动，所以：

- **字号要大** —— hero 文字至少 80px 起，远观也能看清
- **标题要有层次和强调** —— 长标题不要整段同一种黑色大字；按语义拆成
  大 / 中 / 小层级，并用主题 `--accent` 强调关键词，让观众一眼看到重音
- **留白要多** —— 舞台四边都要让出大留白，画面不要塞满
- **留白不是空白** —— 如果半屏以上没有承载信息，必须补流程、账本、
  仪表、工单、剖面、地图、卡片组、路径线等视觉信息；空白只能用于聚焦，
  不能用于偷懒
- **配色要舒服** —— **颜色和字体家族必须用主题 token**（保证换主题不破）；
  字号 / 间距 / 时长这些章节按内容自由发挥（详见下方「代码层最小约束」）
- **动画要舒服 + 炫酷** —— 出现得干净利落，停下来不抢戏；炫酷靠
  **设计巧思**（内容驱动的演示动画），不靠**速度暴力**或**密集闪烁**
- **动画要贴口播** —— 入场、连线、节点点亮、账本增长等动作要能覆盖口播
  的主要节拍；不要一瞬间全出现，然后让画面在长口播里静止

## 长口播拆页规则

如果一个 step 的口播明显偏长，或用户反馈“这一页口播太长”，必须优先考虑
拆成 2 个或更多 HTML step，而不是让同一页静止等待。

拆页时必须同步：

- 对应章节组件的 step 分支
- `narrations.ts` 的数组长度和文本
- 现有音频文件编号：可重新 TTS，也可用 `ffmpeg` 切分原 mp3 并顺延后续编号
- `npm run extract-narrations`
- `npm run build-master-audio`，如项目使用 normalized audio，也要重建 normalized master
- `presentation/src/hooks/useStepper.ts` 的 `STORAGE_KEY`
- `outline.md` / 项目基线里的 step 数记录（如果项目有）

## 本项目默认视觉锚点：D2 纸面档案编辑风

除非用户明确指定其它主题，每章都优先沿用 `d2-editorial-grid`，并把已经
验收过的第 1 章当作视觉 anchor。

必须稳定保留的视觉语言：

- 暖白纸面 + 极淡网格，不做深色黑板或纯黑背景
- 深棕 / 墨色标题，中文大标题优先使用主题的宋体显示字
- 锈红作为唯一强强调色，少量用于标签、状态、关键线条
- 等宽 metadata、阶段胶囊、档案卡、浏览器窗口、文件页、流程节点等
  “知识管理 / AI coding / 项目复盘”视觉原语
- 不加品牌水印。信息 chrome 只承担导航 / 状态 / 上下文说明，不抢画面
- 画面要像一套整理好的文档工作台，不像通用 SaaS landing page
- **风格统一不等于组件统一**：统一的是纸面、字体、色彩、密度和信息
  chrome；每页主体组件必须由内容关系决定。人物用关系图 / 履历轴，
  对比用天平 / 分栏 / 切片，流程用路径 / 节点线，数据用图表 / 仪表，
  案例用拼贴 / 文件墙。不要把所有内容都塞进同一种 card。
- **层面 / 角色 / 环节必须分镜化**：当一章在介绍不同层面、产业环节、
  公司角色、概念模块时，关键对象要有自己的 HTML 场景或主视觉隐喻。
  不能一整章复用同一张页面，只换标题、active 高亮、侧栏文案或少量小组件。
  参考标准：总览可以是链路账本；应用可以是收款单 / 产品窗口；模型可以是
  API 调度台；云和数据中心可以是工厂 / 机房排班；芯片能源可以是电力转算力
  剖面；需求爆发可以是冲击波 / 压力传导 / 采购清单。
- **HTML 表现力可以夸张**：优先让 CSS / SVG / Canvas / JS 演出“发生了什么”，
  例如账本、票据、流程线、工厂、剖面、仪表、工单墙、采购单、冲击波。
  不要满足于把口播文字放进卡片。

明确不要：

- 黑板风 / chalkboard / terminal-green / 大面积深色舞台
- 蓝紫科技渐变、紫粉对角渐变、冷蓝 dashboard
- 泛用圆角 SaaS 卡片堆叠、彩色左边框列表
- 只把口播文字放大排版，缺少文件、浏览器、流程、索引等可视化物件
- 顶部 phasebar、metadata、browser chrome 互相压住或重叠
- 全屏只有一句大标题，没有辅助事实、结构线索或视觉演示
- 大标题全黑、无关键词强调、无大小层级，导致重音不明显
- 下半屏或大块区域完全空着，没有承担节奏、信息或视觉演示功能
- 后续章节全部复用同一套 `HeroText + Card + InfoStrip` 模板
- 整章所有 step 都是同一张图，只做 active / muted / glow / 文案替换

## 避免 AI 味

AI 生成的网页有几种共有的"视觉指纹"，**全部不要**：

- 紫粉 / 蓝紫对角渐变背景
- 圆角卡片 + 彩色左边框装饰
- 渐变按钮 + 大圆角药丸
- emoji 当图标用
- 假数据 / 假 logo / 假"X 万用户"
- 整章 N 步用同一种入场动画（全场 fade / 全场 blur）
- 每步都挂 ken burns / 光晕呼吸 / 持续闪烁
- 每屏右下角都挂 mono 角标 / 序号

缺的东西**承认缺** —— 用 placeholder 占位卡（一张写着"image · 16:9
描述"的卡片，按真实比例留位）。**不要**用 emoji 凑、不要找无关图凑、
不要编数字。**没有就承认没有**，比 fake 强一百倍。

## 框架已经搭好的部分（理解就好，不需重写）

- **16:9 固定舞台**：内容设计在 1920×1080 上，外层 transform scale
  缩到任何视口，外围 letterbox 留黑 —— **没有响应式断点**
- **舞台居中 + 大留白**：上下左右四边都让出至少 80px 的安全区
- **隐形进度条**：屏幕底部默认完全透明，鼠标悬到底部边缘才出现，
  支持点击跳转章节（录屏时摄像头看不到任何 chrome 控件）
- **全局 step 驱动**：点击舞台空白处 / 键盘 ←/→ 推进；章节是 `step`
  的纯函数，没有定时器、没有命令式状态
- **每个 step 必须重新触发入场动画**：舞台里的章节容器 key 要包含
  `chapter id + step`，否则同章切页会复用 DOM，CSS 入场动画看起来像没写。

## 代码层最小约束

不能踩的红线，其它怎么写都行：

### 必须用 token（换主题不破的底线）

- **颜色**：`--shell` / `--surface` / `--surface-2` / `--surface-3` /
  `--text` / `--text-2` / `--text-mute` / `--text-faint` / `--rule` /
  `--accent` / `--accent-soft` / `--accent-glow` ——
  **禁硬编码 hex / rgb / 颜色名**
- **字体家族**：`--font-display-cn` / `--font-display-en` / `--font-body`
  / `--font-mono` —— **禁硬编码字体名**
- **主题性格签名**通过 primitive class 自动接入，**不要在章节 CSS 里
  重定义它们**：
  - `.hero-num`（hero 数字风格 —— 主题决定衬线 / 等宽 / 粗黑）
  - `.rule`（分割线 —— 主题决定 1px 实线 / 4px 实线 / 2px 虚线）
  - `.card`（卡片 —— 主题决定圆角 + 阴影性格）
  - `.stage-frame`（舞台底色 / 圆角 / 阴影 / 装饰图案 / vignette
    全自动，章节什么都不用做）

### 可硬编码 / 可 token，按内容自由（解锁章节自由设计）

- **字号**：想要 80px 就写 80px，想用 `var(--t-h1)` 也行
- **间距 / padding / margin**：按画面节奏写具体值
- **动画时长 / 缓动 / keyframe**：按动画意图写具体值
  （**节奏气质**参考 `theme.json` 的 `mood` —— 慢主题别写 200ms 的快动画）
- **边框宽度 / 非性格圆角 / 字距**：随手写
- **gap / grid 布局尺寸**：按画面构图写

### 其它工程红线

- 默认不加新依赖。React + CSS + SVG 足够时保持轻量；只有明确需要复杂图表
  / 图标库 / motion library 时，才允许增加依赖，并同步更新 `package.json`
  与验证命令。
- 复杂章节可以在本章 `.tsx` 内拆 `MetricCard` / `SvgGauge` / `FlowMap`
  等小组件，但它们只服务本章，不跨章 import。
- 不用 `setTimeout` / `setInterval` 驱动动画 —— 用 CSS keyframes
- 章节内的可交互元素（按钮 / 自定义控件）加 `data-no-advance`，
  否则点了会被舞台误推进 step
- 章节代码物理隔离：每章独立文件夹、独立 CSS 类前缀，不跨章 import
- **每章必须有 `narrations.ts`**（与 `<Chapter>.tsx` 同目录）：
  - 数组长度 **=** 章节代码里 `if (step === N)` 出现的最大 N + 1
  - 每个元素 = 一个 string，该 step 要播的口播文本（来自 `script.md`
    对应段，**语义一致**——可微调标点 / 断句以适配 TTS，但不能漏关键短语）
  - 完全无音频的过场 step 用空串 `""`，Auto 模式会按字数估时撑过
  - 这是**音频合成 + Auto 模式自动推进的唯一真相源**，写错或漏写
    会让录屏对不上嘴
- **动画时长必须 ≤ 该 step 的口播时长**——Auto 模式严格按音频结束推进，
  没有"等动画跑完"的兜底。动画太长 → 三选一：**写更长口播 / 拆 step
  / 调动画速度**。详细机制见 [`AUDIO.md`](AUDIO.md)

## 完工自检（写完每章**强制**执行，不可跳过）

> ⚠️ **硬性流程**：章节实现完成后**必须**走完下面的自检 → 修复 → 汇报
> 三步。**禁止**"实现完成 → 直接汇报给用户"。
>
> **执行方式**（按能力降级）：
>
> 1. **优先 Agent Teams**：开一个独立的 reviewer agent，传入本章代码路径
>    + 本文件 Part「完工自检」清单，让它**逐项核查 + 出结论**（哪几条
>    pass / 哪几条 fail + 证据）。
> 2. **其次 subAgent**：当前 agent 没有 Teams 能力但能开 subagent，用 subagent
>    走同样的流程。
> 3. **都没有**：当前 agent 自己**严格逐项**核查，不允许目测一遍就放行。
>
> **拿到自检结论后**：先按 fail 项**改完代码**，然后再向用户汇报"做完
> 了 + 自检结论 + 改了什么"。**直接拿原始结论汇报但不修复 = 违规**。

写完一章 + 在浏览器点完一遍后逐项过：

- [ ] **每章至少 1~2 处 CSS / SVG / Canvas / JS 视觉演示** —— 没有 = 回去补
- [ ] **每章至少一个富 HTML 主视觉**：SVG 图表 / 仪表 / 流程线 / browser
      mock / terminal mock / file tree / metric dashboard / document wall 等，
      不能只有大标题、正文和普通卡片
- [ ] 数字 / 流程 / 对比 / 状态类内容已优先考虑
      [`RICH-HTML-TOOLKIT.md`](RICH-HTML-TOOLKIT.md) 的 React + SVG primitives
- [ ] 概念旅程 / 工具链 / 输入输出 / 流程类内容已优先考虑
      **CSS 档 SVG 分镜小剧场**：有小人、有路径、有节点状态、有镜头推进，
      不是静态插画或卡片堆
- [ ] SVG 路径线、窗口边框、波形、图表线条已尽量用 `stroke-dashoffset`
      自绘；没有只靠 fade in 冒出来
- [ ] 关键流程节点有 `active` / `done` / `muted` 三态：当前节点清楚，
