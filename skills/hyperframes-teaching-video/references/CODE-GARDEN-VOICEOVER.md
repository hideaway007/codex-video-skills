# code秘密花园式技术口播风格

来源：本地整理的 Code Garden 口播参考文稿。

这份归纳基于最近十条转写稿。原文是 Whisper `small` 转写，存在术语错字、
标点漂移和人名/产品名误识别，所以只能作为风格样本，不能作为事实证据。

## 使用边界

这是 `web-video-presentation` 的唯一口播写作参考，用于 Phase 1 生成
`script.md`。写稿时只读这一份，不再读取其它写作规则。

> **关键优先级**：这份风格层不是让稿子变成“完整讲义”。它要把文章
> 改成“边看屏幕边讲”的技术教程口播。信息保留必须通过具体场景、屏幕
> 推进和个人判断来完成，不能只是把原文观点按顺序总结一遍。

## 样本概况

- 主题集中在 AI coding、Agent、Skill、MCP、RAG、Harness、模型工具和本地配置。
- 视频时长大多在 12-23 分钟，一条接近 29 分钟，属于中长教程型内容。
- 内容重点是“讲清楚一个工具/概念为什么重要，以及怎么用”。
- 常见目标受众：正在用 AI 工具做开发、自动化、内容生产或知识管理的人。
- 讲法偏“工程实践 + 真实演示 + 个人判断”，不是官方文档复述。

## 核心结构

### 1. 开头：先给现象或结果

常见开法：

- 最近大家都在问的一个现象
- 一个已经跑起来的成果
- 一个行业热词但很多人没讲清楚
- 一个真实痛点：配置难、效果不稳、成本高、结果不可控

写稿时不要先讲定义。先让观众知道“这事跟我有什么关系”。

### 2. 30 秒内交代路线图

可以给“三件事”或“几个部分”，但要口语化：

- 先看它到底解决什么问题
- 再看背后的机制
- 最后带大家走一遍实操

路线图只服务一件事：让长视频不散。

### 3. 先交底，再解释

这类稿子经常先把结论或真实取舍讲出来。不要写“不是 X，而是 Y”或
“不是 A，是 B”，直接说正向判断：

- 它用网页录屏做视频
- 真正卡住的是外部运行系统
- 这是一套可复用的工作流约束

这种“先交底”会让后面的教程更可信。

### 4. 用误区对照解释概念

高频表达模式：

- “很多人会以为是 X，但真正影响结果的是 Y。”
- “表面上看是工具问题，实际上是流程问题。”
- “模型是有能力的，但你需要一套机制驾驭这个过程。”

适合解释 Agent、Harness、Skill、RAG、Context、Cache 这类抽象概念。

### 5. 例子密度高

每讲完一个抽象概念，马上接一个具体例子：

- 让 Agent 处理 PDF、写 Excel、生成网页
- 从文章到 `script.md`、`outline.md`、网页章节、TTS、录屏
- 搜索文件、读取知识库、调用 API、修复失败步骤

原则：不要只说“能力很强”，要说“我让它做了什么，它出来了什么结果”。

### 6. 实操段落是“看到什么就解释什么”

教程进入实操后，不要只列步骤。每一步都要有：

- 现在做什么
- 屏幕上应该看到什么
- 这说明什么
- 如果没看到，可能是什么问题

这样更像真实演示，不像安装文档。

### 7. 经常从系统视角收束

视频后半段常把具体工具上升到机制：

- 上下文怎么管理
- 工具怎么接入
- 状态和记忆怎么保存
- 评估和观测怎么做
- 失败后怎么恢复
- 安全边界在哪里

这会让教程不只是“怎么装”，而是“为什么这样设计”。

### 8. 结尾给具体动作

常见结尾包括：

- 仓库或链接放评论区
- 可以拿自己的文章/项目试一下
- 下期继续讲实战或对比
- 如果有问题去 issue 或评论区

项目默认写稿时可以保留具体 CTA，但不要复刻固定三连口号。

## 语气特征

- 第一人称强：我试过、我做了一个、我之前也踩过。
- 第二人称自然：你会发现、你直接把它丢进去、你不用每次重新讲规则。
- 解释时有判断：这个方案可控、这个很麻烦、这个风险很高。
- 对新概念不神化：会讲限制、成本、安全、配置门槛和翻车点。
- 必须有适量口头推进词：那、然后、我们来看、这里可以看到、这里需要注意、问题来了。
- 只修掉机械重复，不要把口头推进词全部清掉；否则会变成讲义稿。

## 三个硬性风格动作

生成 `script.md` 时，每一章至少做一次下面三类动作：

1. **演示感**：把抽象观点落成屏幕动作。常用句式：
   - “我们来看这张图。”
   - “这里可以画成两列。”
   - “如果做成网页画面，我会把 X 放中间。”
   - “你注意右边这个指标。”
2. **个人判断**：给出讲者自己的取舍，不只转述文章。常用句式：
   - “我觉得这句话才是重点。”
   - “这个地方最容易被低估。”
   - “我的建议是先别急着做 X。”
   - “这一步已经是最低门槛。”
3. **具体场景**：每个抽象概念后面接一个可想象的任务或失败现场。常用方向：
   - 一个真实工具：Claude Code / Claude Cowork / GitHub / PDF / Excel / dashboard
   - 一个具体文件：`CLAUDE.md`、scope 文档、指标报告、访谈清单
   - 一个失败画面：新 session 重新发明架构、按钮能点但没人要、创始人审批队列堵住

如果一章只有观点总结，没有屏幕推进、个人判断和具体场景，判定为“不像
code秘密花园”，必须重写。

## 常用句式骨架

可以借用结构，不要固定照抄：

- “先跟大家交个底，真正影响结果的是 Z。”
- “那问题来了，怎么让它每次都稳定做出来？”
- “你直接跟 Agent 说 X，它可能也能做，但很容易跑偏。”
- “这一步很关键，因为后面所有流程都会依赖它。”
- “如果只是做一个小 demo，一个对话可能够了；但如果要做真实项目，就不一样了。”
- “你会发现，真正要解决的是整条链路能不能稳定跑。”
- “所以这个东西表面上看是 X，最后会落到 Y。”

## 生成 `script.md` 时的执行顺序

1. 先找素材里最值得讲清楚的问题，不要按原文目录机械复述。
2. 开头直接给结果、冲突、痛点或可观察现象，不先写百科定义。
3. 30 秒内让观众知道这期解决什么，随后进入具体例子或实操。
4. 每个抽象概念后面接一个真实任务、失败场景或屏幕结果。
5. 实操段按“操作 -> 看到什么 -> 为什么重要 -> 下一个动作”推进。
6. 后半段补机制、边界、风险、失败处理，不只停留在步骤。
7. 结尾给具体下一步：仓库、命令、评论区问题或下一期主题。

### 防止生成“讲义稿”的额外检查

- 不要把 `---` 理解成“每个短句都独立成块”。`---` 是节奏边界，不是
  把口播切碎成提纲卡片。一个自然 step 可以有 2-4 句连续口播。
- 不要为了追求短句，把“那 / 然后 / 我们来看”这类真人讲解过渡全删掉。
- 不要只按原文章节顺序复述。每个阶段都要转换成“观众在屏幕上看到什么、
  主播怎么判断、用户如何照做”。
- 信息保留率不是“观点覆盖率”。关键事实要保留，但表达方式必须口播化、
  场景化、画面化。

## 需要避免

- 不要复制账号身份、固定问候语和口号。
- 不要把每段都写成“首先、其次、最后”；长视频可以有结构，但要自然过渡。
- 不要让“然后 / 那 / 所以呢”成为每一句的开头；但也不能完全不用。
- 不要堆“底层逻辑、本质上、颠覆认知”这类 AI 腔。
- 不要把概念讲成百科定义。每个定义后面都要跟一个任务例子。
- 不要为了像教程而罗列命令。口播讲动作和判断，屏幕画面展示命令。
- 不要沿用转写稿错字：`Scale` 可能是 `Skill`，`Cloud` 可能是 `Claude`，`Gatehub` 应为 `GitHub`。

## 快速改写检查

拿到一段初稿后，按下面顺序改：

1. 第一段有没有具体结果、冲突或痛点？
2. 观众 30 秒内知不知道这期要解决什么？
3. 有没有“不是 X，而是 Y / 不是 A，是 B”的万能转折？有就改成直接判断。
4. 每个抽象词后面有没有例子？
5. 实操步骤有没有“看到什么”和“为什么重要”？
6. 后半段有没有边界、风险或失败处理？
7. 结尾有没有具体下一步？
8. 术语有没有核对过真实写法？
9. 每章有没有至少一次屏幕推进、个人判断和具体场景？
